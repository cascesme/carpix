    Method signature: async def get_or_fetch(self, brand: str, model: str, year: int) -> FileResponse.

    Step 1 — Normalize: derive brand_key = canonical_key(brand), model_key = canonical_key(model). Use these normalized keys for all DB lookups, storage paths, and lock keys.

    Step 2 — Per-key lock acquisition: the lock key is the tuple (brand_key, model_key, year). Check self._locks for this key; if absent, create a new asyncio.Lock() and store it. Acquire the lock with async with self._locks[key]:.

    Step 3 — Inside the lock, implement cache-aside:
      a. Call entry = await self._repo.find(brand_key, model_key, year).
      b. If entry is not None: check whether Path(entry.local_path).exists(). If the file exists, return self._storage.file_response(brand_key, model_key, year). If the file does NOT exist (self-healing path), fall through to the fetch steps below rather than returning a 500.
      c. If entry is None or the file was absent (self-healing): call url = await self._wikimedia.find_jpeg_url(brand, model, year). Note: pass the original (un-normalized) brand and model strings to find_jpeg_url, because the Wikimedia search query benefits from the original casing (e.g. "Toyota Corolla" vs "toyotacorolla"). If url is None, raise HTTPException(status_code=404, detail="No image found for this vehicle").
      d. Download the image: use httpx.AsyncClient() as an async context manager, perform an httpx GET request to url with a 30-second timeout (httpx.Timeout(30.0)). Call response.raise_for_status(). Capture image_bytes = response.content.
      e. Save: saved_path = await self._storage.save(brand_key, model_key, year, image_bytes).
      f. Extract file_title from url: use url.rsplit("/", 1)[-1] as the file title string.
      g. Insert: await self._repo.insert(brand_key, model_key, year, local_path=str(saved_path), source_url=url, file_title=file_title).
      h. Return self._storage.file_response(brand_key, model_key, year).

    The lock must wrap steps a through h entirely, so a second concurrent waiter that enters the lock after the first completes will find the DB row already inserted and return the cached file immediately.

However, if the tests mock httpx at the transport level via respx_mock, the implementation should create its own httpx.AsyncClient() internally in step d. Import asyncio, Path from pathlib, httpx, HTTPException from fastapi, FileResponse from fastapi.responses, canonical_key from carpix_images.domain.normalize.

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| ImageService → Wikimedia CDN | thumburl is an unvalidated external URL; ImageService issues an httpx GET to this URL — SSRF vector |
| ImageService → filesystem | saved_path is derived from canonical_key-normalized inputs; traversal guard enforced by StorageService._validated_path |
| concurrent callers → asyncio.Lock | two concurrent coroutines for the same key must serialize through one lock to prevent double-fetch and double-write races |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-05-02-S | Spoofing | ImageService httpx download | accept | thumburl sourced from Wikimedia Commons API (trusted upstream); no user-supplied URLs downloaded |
| T-05-02-T | Tampering | saved image bytes | accept | bytes written directly from Wikimedia CDN response; no mutation; StorageService path is canonical |
| T-05-02-R | Repudiation | cache entry insert | accept | Internal service, no auth layer in v1; audit logging deferred to OBS-02 (v2) |
| T-05-02-I | Information Disclosure | FileResponse path | mitigate | StorageService._validated_path enforces is_relative_to(base_dir) before serving any path (STORE-02, already implemented) |
| T-05-02-D | Denial of Service | per-key asyncio.Lock | mitigate | Lock is released in all paths (async with context manager guarantees release even on exception); prevents unbounded lock hold |
| T-05-02-E | Elevation of Privilege | SSRF via thumburl | mitigate | thumburl comes only from WikimediaClient which fetches from commons.wikimedia.org API — user input never directly used as download URL; canonical_key normalization strips path traversal chars from brand/model before constructing local paths |
| T-05-SC | Tampering | pip installs | accept | No new packages introduced in Phase 5; all dependencies already installed and audited in prior phases |
</threat_model>
